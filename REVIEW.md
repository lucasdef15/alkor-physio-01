# REVIEW — Revisão completa do produto ΛLKOR Physio

Revisão realizada sobre todo o projeto (páginas, layouts, componentes, hooks, animações, SVGs, backgrounds e arquitetura), avaliando o site como um template comercial premium da Alkor Labs para fisioterapeutas cardiorrespiratórios.

---

## 1. O que foi encontrado

### Produto / Storytelling
- **A página real contém apenas duas seções** (Hero e "Para quem é"), mas o header, o footer e os CTAs apontavam para seções que não existem: `#especialidades`, `#sobre`, `#depoimentos`, `#como-funciona`, `#agendar`, `#tratamentos` e `#contato`. Todos esses links eram **cliques mortos** — o maior quebra-confiança do produto. Um usuário (ou um cliente comprando o template) clica em "Agendar avaliação" e nada acontece.
- O fluxo desejado (Hero → Para quem é → Especialidades → Sobre → Depoimentos → Contato) só existe nos links de navegação; as seções intermediárias ainda não foram construídas.

### Identidade / Metadados
- `<title>` era **"Create Next App"** e a description **"Generated by create next app"** — visível na aba do navegador, em compartilhamentos e no Google. Incompatível com qualquer produto comercial.
- `<html lang="en">` em um site 100% em português — errado para leitores de tela e SEO.

### Design system
- A linguagem visual (teal/sky, glass, rounded-full, sombras em camadas, easing `cubic-bezier(0.45,0,0.2,1)`) está **consistente** entre Hero, ForWhom, Header e Footer — as duas seções pertencem claramente ao mesmo produto.
- Tokens definidos em `globals.css` (`--primary`, `--border`, etc.) são pouco usados; as seções usam a paleta Tailwind diretamente. Não é um problema funcional, mas está documentado como melhoria futura.
- Fonte `Geist Mono` era carregada no layout e **nunca usada** — peso morto no bundle de fontes.

### Acessibilidade
- Botão do menu mobile sem `aria-expanded`/`aria-controls` e com label em inglês ("Toggle menu").
- Menu mobile não fechava com `Escape`.
- CTAs de navegação implementados como `<button>` + `scrollIntoView` em vez de `<a href>` — sem semântica de link, quebravam sem JS.
- Animações do Hero (`scroll-dot`, `badge-breathe`, `animate-ping`) **ignoravam `prefers-reduced-motion`** (a ilustração ForWhom já respeitava).
- Âncoras internas sem compensação para o header fixo — o título da seção ficava escondido atrás do header ao navegar.

### Performance
- O canvas do Hero rodava seu `requestAnimationFrame` **para sempre**, mesmo com o Hero fora da tela — desperdício contínuo de CPU/GPU ao ler o resto da página.
- O canvas ignorava `prefers-reduced-motion` (só desabilitava o parallax).
- O loop da ilustração ForWhom (`useBreathingAnimation`) também rodava fora da tela.

### Código
- `components/ui/Button.tsx` — componente **morto**, sem nenhum uso.
- Lista de links de navegação **duplicada** entre `Header.tsx` e `Footer.tsx` (e divergente entre si).
- Ícone de check SVG repetido 3× inline em `Indicators.tsx`.
- Padrão de link com underline animado repetido 3× no Footer.
- Blocos de comentários "changelog" (Header v2.0 / Footer v2.0) descrevendo diffs antigos — ruído sem valor para quem mantém o código.
- Atributo `data-header-theme="dark"` no Footer sem nenhum consumidor.

---

## 2. O que foi alterado e por quê

### Navegação honesta (maior impacto de confiança)
- **Todos os links agora levam a lugares reais.** A navegação foi reduzida a `Início → Para quem é → Contato` (fonte única em `components/layout/navLinks.ts`, compartilhada por Header e Footer). *Motivo: princípio "prefira remover" — links mortos destroem a confiança; melhor menos itens, todos funcionais. Os itens Especialidades/Sobre/Depoimentos voltam quando as seções existirem.*
- Hero recebeu `id="inicio"`; o Footer recebeu `id="contato"` (é onde vivem telefone, e-mail e o CTA de agendamento).
- CTA primário do Hero ("Agendar avaliação") agora leva a `#contato`; o secundário virou "Conhecer o tratamento" → `#para-quem` (a próxima seção — reforça o storytelling em vez de apontar para uma seção inexistente).
- "Fazer Agendamento" no Footer virou um `mailto:` real (apontar para `#contato` seria rolar para ele mesmo).
- `scroll-behavior: smooth` + `scroll-padding-top: 6.5rem` no `html`: navegação por âncora suave e sem o título escondido atrás do header fixo (com fallback `auto` para reduced motion).

### Identidade / Metadados
- Título e description reais: **"ΛLKOR Physio — Fisioterapia Cardiorrespiratória"**. `lang="pt-BR"`.

### Acessibilidade
- Botão do menu: `aria-expanded`, `aria-controls="mobile-menu"`, label em português que reflete o estado ("Abrir/Fechar menu").
- `Escape` fecha o menu mobile.
- Todos os CTAs de navegação viraram `<a href="#...">` semânticos (funcionam sem JS, aparecem como links para tecnologia assistiva).
- `prefers-reduced-motion` agora desliga também `scroll-dot`, `badge-breathe` e `animate-ping`.

### Performance
- Canvas do Hero: `IntersectionObserver` pausa o loop quando o Hero sai da tela e o retoma ao voltar; com `prefers-reduced-motion`, renderiza **um único frame estático** (a composição continua bonita, sem movimento).
- `useBreathingAnimation`: mesmo padrão — o loop só roda com a ilustração visível.
- Remoção da fonte `Geist Mono` não utilizada.

### Código
- Removido `components/ui/Button.tsx` (morto) e `data-header-theme` (sem consumidor).
- `navLinks.ts` como fonte única de navegação.
- `Indicators.tsx`: itens mapeados de um array (1 SVG em vez de 3 cópias).
- Footer: subcomponente `FooterLink` elimina a triplicação do padrão de underline animado; Footer deixou de ser client component (não precisa mais de handlers).
- Removidos os comentários-changelog de Header e Footer.

---

## 3. Impacto esperado na experiência

- **Confiança:** cada clique responde. A aba do navegador, resultados de busca e compartilhamentos mostram a marca, não "Create Next App".
- **Fluidez:** navegação por âncora suave, com o título da seção corretamente visível abaixo do header.
- **Inclusão:** usuários de teclado e leitores de tela conseguem operar o menu e entender seu estado; quem prefere movimento reduzido tem uma experiência estática digna, não um site quebrado.
- **Bateria/CPU:** os dois loops de animação só trabalham quando estão na tela — relevante em mobile.
- **Manutenção (valor do template):** navegação em um único lugar, menos duplicação, menos código morto — mais barato de customizar por cliente.

---

## 4. O que ainda pode melhorar em versões futuras

1. **Construir as seções faltantes** — Especialidades, Sobre, Depoimentos e um bloco de Contato/agendamento dedicado (formulário ou link de WhatsApp) — e devolvê-las à navegação. É o maior salto de valor possível para o template.
2. **Configuração por cliente:** nome, CREFITO, telefone, e-mail, cidade e claims ("+10 anos de experiência") estão hardcoded; extrair para um arquivo de configuração único (`site.config.ts`) tornaria o template vendável "de fábrica".
3. **Adotar os tokens do design system:** migrar as cores/opacidades ad hoc das seções para os tokens de `globals.css` (ou remover os tokens não usados).
4. **Focus trap no menu mobile** (hoje: Escape + aria; um trap completo seria o próximo nível).
5. **Open Graph/imagem social** e favicon com a marca ΛLKOR.
6. **Scroll-reveal sutil** nas entradas de seção (com reduced-motion), reforçando o ritmo premium entre seções conforme novas seções forem adicionadas.
